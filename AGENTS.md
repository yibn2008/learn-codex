# learn-codex 项目开发规则

## 中文内容输出规则

- **禁止产生 Unicode 乱码**：输出中文内容时，必须确保所有中文字符完整输出，不得出现 U+FFFD 替换字符（显示为 `���`）。这类乱码是中文字符在输出时被截断造成的。
- 每次生成或修改包含中文的文件后，应使用 `grep -rn '���'` 检查是否存在乱码，如果存在则立即修复。

## 项目结构

- `PROJECT.md` — 项目总目标文档，供多 Agent 协作时了解全局
- `PROGRESS.md` — 任务管理文件，追踪章节状态和每轮工作记录
- `wiki/` — Wiki 章节文件（markdown）
- `wiki/images/` — 图表资源
- `diagrams/` — 中间分析产物
- `index.html` — Wiki Web 预览页面
- `.screenshots/` — Playwright 截图存放目录（已 gitignore）
- `.playwright-mcp/` — Playwright MCP 产物（已 gitignore）

## 文件存放规则

- Playwright 截图必须保存到 `.screenshots/` 目录，不要放在项目根目录
- 临时测试产物不要提交到 git

## Wiki 编写规范

- 所有 Wiki 内容使用中文编写
- 遇到 Rust 特有概念时，以「知识点小贴士」内联讲解，不设独立的 Rust 入门章节
- 引用源码时标注文件路径
- 每完成一个章节，更新 `PROGRESS.md` 的状态

## Mermaid 图表规则

### 核心原则：图要能看清

- **单图节点数上限 15 个**。超过就必须拆分成多个图，每个图聚焦一个层面。
- **优先用表格展示详细信息**，图只展示结构和关系。不要把所有细节塞进图里。
- **标签文字要简短**。节点名用短名（如 `cli` 而非 `codex-cli — 命令行路由入口`），详细说明放在图外的正文或表格中。

### 布局选择

| 场景 | 推荐布局 | 说明 |
|------|---------|------|
| 分层架构（调用关系） | `graph TD` | 自上而下，每层一个 subgraph |
| 数据流、管道 | `graph LR` | 左到右，线性流程 |
| 时序交互 | `sequenceDiagram` | 多方交互、请求/响应 |
| 状态变迁 | `stateDiagram-v2` | 状态机、生命周期 |
| 依赖关系 | `graph LR` | 左到右更紧凑，避免 `graph BT` |

### 避免的问题

- **不要用 `graph BT`**（自下而上）：不符合阅读直觉，改用 `graph TD` 或 `graph LR`。
- **不要在一个图中混合多个 subgraph 层级**：Mermaid 对嵌套 subgraph 的布局支持差，容易变形。
- **不要让单行节点横向超过 5-6 个**：否则图会被压扁成一条横线。如果同层节点多，用 `direction LR` 在 subgraph 内横向排列，或拆成多个图。
- **避免大量交叉连线**：如果连线超过 10 条且互相交叉，说明图太复杂了，需要拆分。

### 拆图策略

当一个概念太复杂时，按以下方式拆分：
1. **总览图**：只展示顶层模块和关系（5-8 个节点），建立整体认知
2. **局部详图**：每个模块内部的详细结构，各自一张图
3. **表格补充**：将完整列表、属性等信息用表格呈现，不塞进图里
